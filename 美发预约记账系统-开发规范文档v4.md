# 美发预约记账系统

## 开发规范文档 v4.0

**技术栈：** uni-app (Vue 3) + Node.js + MongoDB + COZE + 飞书  
**输出端：** 微信小程序（顾客/店长）+ Web 管理后台（超级管理员/店长）  
**服务器：** 阿里云 2 核 2G 云服务器  
**日期：** 2026 年 4 月

---

## 1 项目概述

### 1.1 产品定位

美发消费者预约工具 + 美发店管理工具 + 美发产品推荐平台，面向 C 端消费者（主要）和 B 端美发店（辅助），以技术服务商 + 广告推广者角色运营（无需 ICP 证/EDI 证）。

初期策略：先服务 1 家特定美发店打磨产品，验证核心流程后再扩展。

### 1.2 技术栈

| 层 | 技术选型 | 说明 |
|----|---------|------|
| 小程序前端 | uni-app (Vue 3) | 微信小程序，顾客端 + 店长端 |
| Web 管理后台前端 | Vue 3 + Element Plus | 超管和店长的 Web 管理界面 |
| 后端 API | Node.js + Koa | REST API 服务 |
| 数据库 | MongoDB | NoSQL，灵活文档模型 |
| 缓存 | Redis | 会话管理、预约锁、频繁查询缓存 |
| 定时任务 | node-cron | 每日重置预约计数器、自动取消超时预约等 |
| 智能体 | COZE（HTTP 调用 API） | 每个店绑定自己的 COZE 智能体 |
| 管理端/看板 | 飞书多维表格 | 每个店绑定自己的飞书多维表格 API |
| 消息通知 | 微信订阅消息 / 短信（第三方） | 可配置，由店长/超管选择 |
| 部署 | 阿里云 2 核 2G 轻量服务器 | Nginx 反向代理 + PM2 进程管理 |

### 1.3 成本预估

| 项目 | 费用 | 备注 |
|------|------|------|
| 阿里云服务器 | 已有 | 2 核 2G |
| 域名 | 已有 | Vidiu.cn |
| 小程序认证 | 0 元 | 复用已认证公众号，公司主体：河北速播网络科技有限公司 |
| COZE 智能体 | 基础免费 | 每个店自己绑定 |
| 飞书多维表格 | 免费 | 每个店自己绑定 |
| 短信服务（可选） | 按量付费 | 第三方短信 API，如阿里云短信 |
| MongoDB | 免费 | 服务器本地部署 |
| **合计** | **约 0-50 元/月** | 主要取决于短信使用量 |

### 1.4 盈利模式

- **短期（Phase 1-2）：** 产品打磨期，免费给门店使用
- **中期（Phase 3）：** 电商分销佣金（淘宝联盟/多多进宝/抖音精选联盟）
  - 假发：客单价 200-2000 元，佣金 10-20%，单笔 20-400 元
  - 洗护用品：客单价 50-200 元，佣金 10-15%，单笔 5-30 元
- **长期（Phase 4）：** 可选自营产品或增值功能收费

### 2.3 用户信息获取

顾客和店长登录小程序时都需要获得以下微信信息：

- **微信头像**（avatar_url）
- **微信昵称**（nickname）
- **手机号**（phone，通过微信手机号快速验证组件获取）

> 店长还需要在 Web 后台设置账号密码用于 Web 端登录。

### 2.4 角色绑定与入驻流程

#### 2.4.1 顾客注册

自动注册，首次打开小程序自动完成：
```
用户打开小程序 → uni.login() 获取 openid → 
获取用户信息（头像/昵称/手机号）→ 
自动创建 customers 记录
```

#### 2.4.2 店长入驻

```
顾客在小程序「申请成为店长」→ 填写门店信息 → 提交申请 → 
超管在 Web 后台审核通过 → 
自动创建 merchants 记录 + staff 记录（role='owner'）→ 
店长身份生效
```

超管也可直接在 Web 后台手动添加店长。

#### 2.4.3 店员添加（Phase 2）

由店长在 Web 后台或小程序中添加，Phase 1 不做。

## 3 核心功能模块

### 3.1 顾客端功能（微信小程序）

#### 3.1.1 门店浏览
- 查看门店信息、营业时间、服务项目及价格
- 浏览门店首页展示

#### 3.1.2 查询店主空档
- 选择发型师（如果有多个）和日期
- 展示该发型师当日可用的空闲时间段
- 需考虑打烊/段打烊的时间段排除

#### 3.1.3 预约管理
- **创建预约：** 选择服务 → 选择发型师 → 选择日期 → 选择时间 → 确认
- **修改预约：** 修改时间/服务（限制：仅 pending/confirmed 状态可修改，且需在服务前一定时间以上）
- **取消预约：** 取消自己的预约（限制：仅 pending/confirmed 状态可取消）
- **查看历史预约：** 查看所有历史预约记录及状态

#### 3.1.4 个人中心
- 微信头像、昵称、手机号展示
- 消费记录、累计消费金额
- 顾客备注（自己维护的偏好备注）

### 3.2 店长端功能（微信小程序）

> 店长与顾客使用同一个小程序，店长登录后在个人中心看到更多管理功能。

#### 3.2.1 店务管理
- **服务项目管理：** 添加/编辑/上下架服务项目，修改价格、时长、阶段
- **营业时间管理：** 修改营业时间、延长营业时间
- **打烊管理：**
  - 临时打烊：设置某天全天不可预约
  - 段打烊：设置某天的某段时间不可预约（如午休 12:00-14:00）

#### 3.2.2 预约管理
- **查看所有预约：** 按日期/状态筛选，查看全部预约
- **代预约：** 为电话预约的顾客（尤其是老年顾客）代创建预约
  - 输入顾客姓名、电话（可选）
  - 选择服务、发型师、时间
  - 系统自动检查冲突
- **管理顾客预约：** 取消/修改任意顾客的预约

#### 3.2.3 记账管理
- **手动记账：** 在小程序内直接填写收入记录
  - 选择/输入顾客名称
  - 填写金额、支付方式
  - 添加收入明细（多个服务项目）
  - 可关联已有预约

#### 3.2.4 消息通知
- **群发通知：** 向所有顾客或指定顾客群发消息
  - 支持微信小程序订阅消息
  - 支持短信通知（需配置第三方短信 API）
  - 通知渠道由店长/超管在 Web 后台配置

#### 3.2.5 顾客管理
- 查看顾客列表
- 管理顾客备注（店家记录的发型偏好等）
- 查看顾客消费历史

#### 3.2.6 营收统计
- 查看日/周/月营收统计
- 查看服务项目营收排行

### 3.3 COZE 智能体（豆包 App）

每个店绑定自己的 COZE 智能体，店长通过豆包 App 语音输入完成操作。

**设计原则：** 除手动记账（小程序端补充）外，所有管理功能都要通过 COZE 智能体语音方式实现一遍，实现高效的自然语言管理。

#### 3.3.1 核心能力

| 能力 | 说明 |
|------|------|
| 语音记账 | 服务完成后语音记录收入 |
| 预约状态变更 | 开始服务、完成服务、标记未到、取消预约 |
| 散客登记 | 临时到店顾客快速登记 |
| 代预约 | 语音代顾客创建预约 |
| 数据查询 | 查询营收统计和员工业绩 |
| 打烊管理 | 语音设置临时打烊/恢复营业 |
| 群发通知 | 语音触发群发消息 |

#### 3.3.2 为什么选 COZE

- **响应速度快：** 豆包 App 原生语音，秒级响应
- **零/极低成本：** COZE 基础功能免费
- **场景适配好：** 精确配置意图识别和命令映射
- **7×24 可用：** 独立服务，不依赖开发平台
- **每店独立：** 每个店绑定自己的 COZE 账号和智能体，安全隔离

> WorkBuddy 是开发工具，用于系统开发维护，不适合作为生产环境的语音服务。

### 3.4 Web 管理后台

超管和店长通过浏览器访问，用于复杂配置和数据管理。

#### 3.4.1 超级管理员功能
- 审核店长入驻申请
- 手动添加/禁用店长
- 管理所有门店（查看/编辑）
- 平台级数据统计
- 系统配置管理

#### 3.4.2 店长功能
- 门店数据总览（预约、营收、顾客数）
- 预约/交易记录管理
- 服务项目管理
- 顾客数据管理
- 营收统计报表
- **COZE 智能体绑定配置**
  - 填写 COZE Bot ID
  - 填写 COZE API Key
  - 测试连通性
- **飞书多维表格绑定配置**
  - 填写飞书 App ID、App Secret
  - 填写多维表格 Token
  - 选择要绑定的数据表（默认绑定预约表）
- **消息通知渠道配置**
  - 选择通知方式：微信订阅消息 / 短信 / 两者兼用
  - 短信 API 配置（服务商、AccessKey、签名、模板）
  - 微信订阅消息模板配置

### 3.5 大屏看板（飞书多维表格）

店内电视展示今日预约流水、等候顾客信息、当前服务状态。使用飞书多维表格视图实现，电视打开飞书网页即可。

### 3.6 飞书集成

每个店绑定自己的飞书多维表格，支持双向数据同步：

- **本地 → 飞书（outbound）：** 预约/交易变更时自动同步
- **飞书 → 本地（inbound）：** 监听飞书 Webhook，飞书中编辑数据同步回本地
- **绑定关系可配置：** 在 Web 后台选择哪些数据表与飞书多维表格进行绑定，默认绑定预约表

## 4 数据库设计

基于 MongoDB，共 9 个集合（collections）。

所有集合均包含系统字段：`_id`（ObjectId 自动生成）、`create_time`、`update_time`。

### 4.1 admins（超级管理员表）

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|:----:|------|
| username | String | 是 | 登录用户名 |
| password_hash | String | 是 | 密码哈希（bcrypt） |
| real_name | String | 是 | 真实姓名 |
| is_active | Boolean | 是 | 是否启用，默认 true |
| create_time | Date | 自动 | 创建时间 |
| update_time | Date | 自动 | 更新时间 |

> 超管账户在系统初始化时手动创建，不支持自助注册。

### 4.2 merchants（商户/门店表）

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|:----:|------|
| merchant_id | String | 是 | 商户唯一标识，自动生成 |
| name | String | 是 | 门店名称 |
| address | String | 否 | 门店地址 |
| phone | String | 是 | 联系电话 |
| business_hours | Object | 是 | 默认营业时间 {start: '09:00', end: '21:00'} |
| status | String | 是 | 枚举：pending / active / inactive / rejected |
| description | String | 否 | 门店简介 |
| cover_image | String | 否 | 门店封面图 URL |
| owner_id | String | 是 | 店长对应的 users 记录 ID |
| daily_counter | Number | 是 | 当日预约序号计数器，每日凌晨重置为 0 |
| counter_date | String | 是 | 计数器日期 'YYYY-MM-DD'，用于判断是否需要重置 |
| coze_config | Object | 否 | COZE 配置，见 4.2.1 |
| feishu_config | Object | 否 | 飞书配置，见 4.2.2 |
| notify_config | Object | 否 | 消息通知配置，见 4.2.3 |
| sync_config | Object | 否 | 飞书同步配置，见 4.2.4 |
| create_time | Date | 自动 | 创建时间 |
| update_time | Date | 自动 | 更新时间 |

#### 4.2.1 coze_config（COZE 智能体配置）

```json
{
  "bot_id": "xxx",
  "api_key": "xxx",
  "api_endpoint": "https://api.coze.cn/v1/..."
}
```

#### 4.2.2 feishu_config（飞书配置）

```json
{
  "app_id": "cli_xxx",
  "app_secret": "xxx",
  "table_tokens": {
    "appointments": "tblxxx",
    "transactions": "tblyyy"
  }
}
```

#### 4.2.3 notify_config（消息通知配置）

```json
{
  "channel": "wechat_subscribe",  // 枚举：wechat_subscribe / sms / both
  "sms_config": {
    "provider": "aliyun",
    "access_key_id": "xxx",
    "access_key_secret": "xxx",
    "sign_name": "xxx",
    "template_code": "xxx"
  },
  "wechat_template_ids": {
    "appointment_confirm": "xxx",
    "appointment_remind": "xxx",
    "cancel_notice": "xxx"
  }
}
```

#### 4.2.4 sync_config（飞书同步配置）

```json
{
  "enabled": true,
  "bindings": [
    { "collection": "appointments", "table_token": "tblxxx", "enabled": true },
    { "collection": "transactions", "table_token": "tblyyy", "enabled": false }
  ]
}
```

> 默认绑定预约表（appointments），其他表由店长/超管在 Web 后台按需开启。

### 4.3 users（用户表）

统一存储所有微信登录用户（顾客 + 店长 + 后期店员）。

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|:----:|------|
| openid | String | 是 | 微信 openid，唯一标识 |
| union_id | String | 否 | 微信 unionid（多端统一用） |
| nickname | String | 是 | 微信昵称 |
| avatar_url | String | 是 | 微信头像 URL |
| phone | String | 是 | 手机号 |
| real_name | String | 否 | 真实姓名（用户填写） |
| role | String | 是 | 枚举：customer / owner / staff（Phase 1 只有 customer/owner） |
| merchant_id | String | 否 | 所属商户 ID（店长/店员才有） |
| customer_note | String | 否 | 顾客备注（顾客自己写的，如偏好） |
| merchant_note | String | 否 | 店家备注（店主记录的，如发型偏好） |
| visit_count | Number | 是 | 到店次数，默认 0 |
| total_spending | Number | 是 | 累计消费金额（分），默认 0 |
| last_visit_time | Date | 否 | 最近到店时间 |
| create_time | Date | 自动 | 创建时间 |
| update_time | Date | 自动 | 更新时间 |

> - `customer_note` 由顾客在小程序个人中心维护
> - `merchant_note` 由店长在小程序或 Web 后台维护
> - 同一个微信用户首次进入小程序自动注册为 customer，申请店长通过后 role 更新为 owner

### 4.4 services（服务项目表）

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|:----:|------|
| service_id | String | 是 | 服务唯一标识 |
| merchant_id | String | 是 | 所属商户 ID |
| name | String | 是 | 服务名称，如 '女士剪发' |
| category | String | 是 | 分类：cut / perm / dye / care |
| price | Number | 是 | 价格（分），如 3000 = 30 元 |
| total_duration | Number | 是 | 总时长（分钟） |
| staff_busy_duration | Number | 是 | 发型师忙碌时长（分钟） |
| stages | Array | 是 | 服务阶段数组，见 4.4.1 |
| description | String | 否 | 服务描述 |
| is_active | Boolean | 是 | 是否上架，默认 true |
| sort_order | Number | 是 | 排序权重，越小越靠前 |
| create_time | Date | 自动 | 创建时间 |
| update_time | Date | 自动 | 更新时间 |

#### 4.4.1 stages（服务阶段数组）

每个阶段对象结构：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|:----:|------|

#### 4.6.1 status（状态流转）

```
pending → confirmed → in_progress → completed
pending → confirmed（5分钟超时自动确认）
pending → cancelled（顾客/店长取消）
confirmed → in_progress（店长点击开始服务，可微调时长）
confirmed → cancelled（顾客/店长取消）
confirmed → no_show（超时未到）
in_progress → completed（完成服务并记账）
```

| 状态 | 说明 | 触发方式 |
|------|------|----------|
| pending | 待确认 | 顾客创建预约 / 代预约 / 修改预约后重置 |
| confirmed | 已确认 | 店长手动确认 / **5分钟超时自动确认** |
| in_progress | 服务中 | 店长在小程序/COZE 点击开始服务（可微调时长） |
| completed | 已完成 | 店长完成服务并提交记账 |
| cancelled | 已取消 | 顾客或店长取消（双方均可操作） |
| no_show | 未到店 | 店长标记 / 超时未到自动标记 |

> **自动确认规则：** 预约创建后，如果店长在 **5 分钟内** 未操作（未确认/取消），系统自动将状态改为 confirmed，并向顾客推送"预约已确认"通知。
>
> **取消规则：** 顾客和店长均可取消预约。取消后的订单**不保留**在"我的预约"列表中，顾客可重新选择时间新建预约。参考滴滴模式——如果店长因临时有事无法接单，可通过订单上的拨号按钮联系顾客沟通，达成一致后由任一方取消。
>
> **修改预约规则：** 仅 pending/confirmed 状态可修改。修改后状态变为 pending（等待重新确认，再次进入5分钟自动确认倒计时）。已开始服务（in_progress）的预约不可修改。

## 5 预约冲突检测算法

### 5.1 核心原理

只检查 timeline 中 `staff_busy=true` 的阶段，忽略空闲阶段。同时排除打烊时段。

当新预约的任意忙碌时间段与已有预约的忙碌时间段重叠时，判定为冲突。

### 5.2 时间重叠判断

```javascript
function isOverlap(start1, end1, start2, end2) {
  return start1 < end2 && start2 < end1;
}
```

### 5.3 完整检测逻辑

```javascript
async function checkStaffAvailability(staff_id, merchant_id, date, busy_start, busy_end) {
  // 1. 检查是否在营业时间内（含延长营业判断，见 6.5 节）
  const hours = await getEffectiveBusinessHours(merchant_id, date);
  if (busy_start < hours.start || busy_end > hours.end) {
    return { available: false, reason: '超出营业时间' };
  }

  // 2. 检查是否在打烊时段内
  const closedPeriods = await getClosedPeriods(merchant_id, date);
  for (const period of closedPeriods) {
    if (period.type === 'full_day') {
      return { available: false, reason: '当天全天打烊' };
    }
    if (isOverlap(busy_start, busy_end, period.start_time, period.end_time)) {
      return { available: false, reason: '该时段打烊休息' };
    }
  }

  // 3. 检查与已有预约的冲突
  const appointments = await db.collection('appointments')
    .find({
      staff_id,
      date,
      status: { $in: ['confirmed', 'in_progress'] }
    }).toArray();

  for (const apt of appointments) {
    for (const stage of apt.timeline) {
      if (!stage.staff_busy) continue;
      if (isOverlap(busy_start, busy_end, stage.start, stage.end)) {
        return { available: false, conflict_with: apt.appointment_id };
      }
    }
  }

  return { available: true };
}
```

### 5.4 预约创建流程

1. 顾客在首页点击服务项目卡片 → 进入创建预约页
2. 顾客选择日期 → 获取该日可用时间段（自动排除打烊时段和已占用时段）
3. 顾客选择时间段 → 提交预约 → 后端再次校验（防止并发）
4. 校验通过 → 原子生成 appointment_id → 生成 timeline → 写入数据库（status = pending）
5. 向**店长**发送新预约通知（微信订阅消息）
6. 启动 5 分钟自动确认倒计时
7. 店长操作或超时后，向顾客发送结果通知（确认/拒绝）

## 5.5 预约全生命周期流程（核心业务流）

> 整体流程类似滴滴打车：顾客下单 → 店主接单 → 到店服务 → 完成记账。店主通过小程序/COZE 控制订单状态流转。

### 5.5.1 流程概览

```
顾客预约 → 店长收到通知 → 店长操作（确认/取消/拨号沟通/不操作）
               ↓                              ↓
           自动确认                      5分钟超时
               ↓                              ↓
         顾客收到确认通知 ←───────────── 自动确认 confirmed
               ↓
         顾客到店
               ↓
          店长微调时长（可选）
          ↓                   ↓
        不调整              调整时长
          ↓                   ↓
          └───────┬───────────┘
                  ↓
          点击"开始服务" in_progress
          + 通知后续排队顾客（可能推迟）
                  ↓
          完成服务 + 记账 completed
          （输入金额 + 支付渠道）
```

### 5.5.2 店主通知与快捷操作

预约创建后，系统向店长发送**微信订阅消息**通知，通知内容包含：
- 顾客昵称、预约服务、预约时间
- 点击通知可打开小程序跳转到该预约详情页

**预约详情页提供快捷操作按钮：**

| 按钮 | 操作 | 说明 |
|------|------|------|
| ✅ 确认 | pending → confirmed | 手动确认预约 |
| 📞 拨号 | 拉起电话拨号界面 | 快速拨打顾客电话，用于沟通调整时间或取消 |
| ❌ 取消 | pending → cancelled | 取消预约并通知顾客 |

> **设计理念：** 参考滴滴打车模式，不设"拒绝"功能。店长有事时通过拨号按钮直接联系顾客沟通，双方达成一致后由任一方取消即可。

### 5.5.3 5分钟自动确认机制

- 预约创建后启动 **5 分钟倒计时**
- 倒计时期间，店长可操作确认/拒绝/改时间
- **倒计时结束后自动确认**，无需店长操作
- 店长也可在收到通知后直接忽略（不操作），等同于确认

> **实现方式：** 定时任务每分钟扫描 pending 状态的预约，检查 `create_time + 5min < now()` 的记录，自动更新为 confirmed。

### 5.5.4 开始服务前的微调

到店后，店长可能发现实际情况与预约不一致（如长发顾客、发型复杂度不同等），需要微调服务时长：

- 店长在预约详情页可**修改实际服务时长**（如 25分钟 → 30分钟）
- 微调后更新 `actual_duration` 字段和 `timeline` 数组
- **级联影响：** 如果微调后导致后续预约时间冲突，系统仅向受影响的后续预约顾客发送通知，提示"前面的服务延长，您的预约可能推迟"，**不自动调整后续预约时间**
- 如果不微调，直接点击"开始服务"即可

### 5.5.5 完成服务与记账

服务完成后，店长提交记账信息：

- **金额输入：** 输入本次服务总金额（可修改默认价格）
- **支付渠道选择：**

| 渠道 | 说明 |
|------|------|
| 微信支付 | 扫码或顾客微信转账 |
| 支付宝 | 顾客支付宝转账 |
| 现金 | 现金收银 |
| 储值卡 | 会员储值余额扣款（Phase 2） |
| 次卡 | 会员次卡核销（Phase 2） |

- 提交后，预约状态变为 completed，同时创建一条 transactions 记录
- 可通过小程序端操作或 COZE 语音完成（"7号服务完毕，记账30元，微信支付"）

## 6 打烊与营业时间管理

### 6.1 两种打烊模式

| 模式 | 说明 | 示例 |
|------|------|------|
| 临时打烊（full_day） | 某天全天不可预约 | "今天有事休息" |
| 段打烊（time_range） | 某天某段时间不可预约 | "中午 12:00-14:00 午休" |

### 6.2 打烊与预约的交互规则

- 设置打烊后，该时段内的**新预约请求被拒绝**
- 进行打烊操作时，提供复选框选项：
  - ☑️ **取消该时段所有预约** — 勾选后自动将该时段内的预约状态改为 cancelled
  - ☑️ **向已预约顾客发送通知** — 勾选后向受影响的顾客推送打烊通知（根据 notify_config 配置的渠道）
- 打烊信息在**顾客小程序端展示**，告知顾客该时段不可预约
- 已有预约如未选择自动取消，则不受影响，由店长自行联系处理

### 6.3 延长营业时间

用于节假日等客流高峰时段（如春节前、二月二龙抬头），临时延长可预约的时间范围。

**使用场景示例：**

> 原营业时间 09:00-18:00，春节前客流激增，店长设置延长营业至 21:00。
> 顾客在小程序查询空档时，可选择 18:00-21:00 的延长时段进行预约。

**交互规则：**

- 延长营业时间是对 `business_hours.end` 的**临时覆盖**
- 设置后，预约冲突检测算法使用延长后的结束时间来判断是否在营业时间内
- 顾客小程序端正常展示延长后的可用时段，无需特殊提示
- 可设置生效日期范围（如"1月20日-1月28日延长至21:00"），到期后自动恢复默认营业时间

### 6.5 营业时间判断逻辑（更新）

预约冲突检测时，判断某时段是否在营业时间内的逻辑更新为：

```javascript
async function getEffectiveBusinessHours(merchant_id, date) {
  const merchant = await getMerchant(merchant_id);
  let end = merchant.business_hours.end;

  // 检查是否有延长营业配置覆盖
  if (merchant.extended_hours) {
    const ext = merchant.extended_hours.find(e =>
      date >= e.start_date && date <= e.end_date
    );
    if (ext && ext.extended_end > end) {
      end = ext.extended_end;
    }
  }

  return { start: merchant.business_hours.start, end };
}
```

### 6.6 操作入口

- **小程序端（店长）：**
  - 快速设置今天打烊 / 设置午休时段
  - 设置延长营业时间（选择日期范围和延长到的结束时间）
- **COZE 语音：** "今天临时休息"、"设置午休12点到2点"、"春节延长营业到晚上9点"
- **Web 后台：** 批量设置、查看历史记录、管理延长营业配置

## 7 COZE 智能体设计

### 7.1 架构设计

每个店绑定自己的 COZE 智能体，独立配置、独立 API Key。

```
豆包 App → COZE 智能体（每店独立）→ HTTP 调用 → 后端 API → MongoDB
```

COZE 通过 HTTP 插件调用后端统一 API 接口，Header 中携带该店的 `merchant_id` 和 `api_key`。

### 7.2 命令集定义

| 命令 | 触发示例 | 参数 | 说明 |
|------|----------|------|------|
| start_service | 开始服务7号 | appointment_id, duration(可选) | 开始服务，可微调时长，通知后续顾客 |
| complete_and_bill | 7号顾客服务完毕，记账30元微信支付 | appointment_id, amount, payment_method, items | 完成预约 + 创建交易记录 |
| mark_no_show | 8号没来 | appointment_id | 将预约状态改为 no_show |
| cancel_appointment | 取消9号预约 | appointment_id | 将预约状态改为 cancelled |
| walk_in | 临时来个顾客剪发，预计25分钟 | service_name, duration, customer_name(可选) | 创建散客预约，直接 in_progress |
| create_transaction | 记账50元，微信支付 | amount, payment_method, items, customer_name(可选) | 独立记账（不关联预约） |
| proxy_booking | 帮张姐预约明天下午3点剪发 | customer_name, date, time, service_name | 代预约 |
| query_stats | 今天营业额多少 | date_range(可选) | 查询营收统计 |

| extend_hours | 春节延长到晚上9点 / 明天延长到8点 | date_range, extended_end | 延长营业时间 |
| send_notification | 群发通知明天营业时间调整 | message_content, target(可选) | 群发通知 |
| query_available | 明天下午有什么空档 | date, time_range(可选) | 查询空闲时段 |

### 7.3 walk_in（散客）处理逻辑

1. COZE 识别意图为 walk_in，提取参数
2. 根据 merchant_id + service_name 模糊匹配服务项目
3. 调用冲突检测，确保发型师在对应时段可用
4. 创建预约记录：customer_id=null, status='in_progress', source='coze'
5. 生成 timeline，写入数据库，同步飞书

### 7.4 COZE 绑定流程

1. 店长在 COZE 平台创建自己的智能体（或使用模板一键创建）
2. 在智能体中配置 HTTP 插件，指向后端 API 地址
3. 在 Web 后台填写 COZE Bot ID 和 API Key
4. 系统自动将该店的 API 调用路由到对应的 COZE 智能体

## 11 飞书集成设计

### 11.1 每店独立配置

每个店在 Web 后台绑定自己的飞书多维表格：

- 飞书 App ID + App Secret
- 多维表格 Token
- 数据表绑定关系（默认绑定预约表）

### 11.2 同步方向

- **本地 → 飞书（outbound）：** 预约/交易变更时自动同步
- **飞书 → 本地（inbound）：** 飞书 Webhook 触发，监听多维表格变更

### 11.3 同步防循环

飞书数据写入本地时标记 `_sync_source = 'feishu'`，同步到飞书前检查该标记。

### 11.4 同步日志

所有同步操作记录在 sync_logs 集合中，支持失败重试（最多 3 次）。

## 12 安全与权限

### 12.1 数据权限

- 顾客只能查看/操作自己的预约和消费记录
- 店长可以查看/操作本店所有数据

- **小程序：** JWT Token，包含 user_id、role、merchant_id
- **Web 后台：** Session + Cookie，登录后存 session
- **COZE：** Header 传递 merchant_id + api_key
- **飞书 Webhook：** 签名验证

### 12.3 小程序合规

- 类目：先选"工具"类目上线验证
- 小程序名：使用通用名称如"预约助手"
- 营业执照范围包含技术推广服务、广告设计

## 附录 C 预约编号生成示例（原子操作）

```javascript
// 使用 MongoDB findOneAndUpdate 原子操作
async function generateAppointmentId(merchant_id, date) {
  const today = date; // 'YYYY-MM-DD'
  const prefix = today.replace(/-/g, ''); // '20260402'

  // 原子递增 + 每日重置判断
  const result = await db.collection('merchants').findOneAndUpdate(
    {
      merchant_id,
      $or: [
        { counter_date: { $ne: today } },
        { counter_date: { $exists: false } }
      ]
    },
    {
      $set: { counter_date: today, daily_counter: 1 },
      $setOnInsert: { daily_counter: 1 }
    },
    { returnDocument: 'after', upsert: true }
  );

  let sequenceNum;
  if (result.counter_date === today) {
    // 同一天，正常递增
    const incremented = await db.collection('merchants').findOneAndUpdate(
      { merchant_id, counter_date: today },
      { $inc: { daily_counter: 1 } },
      { returnDocument: 'after' }
    );
    sequenceNum = incremented.daily_counter;
  } else {
    // 新的一天，重置为 1
    sequenceNum = 1;
  }

  const seq = String(sequenceNum).padStart(3, '0');
  return { appointment_id: `${prefix}-${seq}`, sequence_num: sequenceNum };
}
```
